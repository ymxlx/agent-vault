# Protocol Specification

The exact schema for every file the agent-vault protocol uses. Read this when you need to validate a file format, parse another agent's output, or extend the protocol cleanly.

## Conventions

- All file paths in this document are relative to the vault root unless stated otherwise.
- All timestamps are ISO 8601, with the vault's configured timezone (default UTC). Format: `YYYY-MM-DD HH:MM` for event lines, `YYYY-MM-DD HH:MM:SS` for frontmatter fields where seconds matter.
- All file IDs (agent IDs, task slugs, decision slugs) are lowercase ASCII with hyphens. No spaces, no underscores in IDs, no unicode. The folder names use these IDs verbatim.
- Frontmatter is YAML, fenced by `---` on its own lines at the top of the file.
- Wikilinks use Obsidian's standard `[[path/to/note]]` syntax. Paths are relative to the vault root and omit the `.md` extension.

## File: `_multi-agent/AGENT_INSTRUCTIONS.md`

Plain markdown. No frontmatter required. Audience: any AI agent that enters the vault. Contains the canonical, tool-agnostic protocol description: entry routine, file formats, event line grammar, concurrency rules, granularity guidance, and tool-specific notes. Written so an agent can read it once at session start and proceed correctly.

This file is generated by `init_vault.py` from `templates/AGENT_INSTRUCTIONS.md` in the skill. If you need to update the protocol across many vaults, edit the template and re-run the script with `--force`; otherwise, edit the vault's copy directly and post a `decided protocol-change` event.

The bridge pointer files at the vault root (`CLAUDE.md`, `AGENTS.md`, `GEMINI.md`) all point at this file. Keeping the canonical content here and the pointers thin means a protocol change only needs to be made once.

## File: `_multi-agent/README.md`

Plain markdown. No frontmatter. Audience: humans who open the vault and want to know what `_multi-agent/` is. It should explain in three or four paragraphs that this folder is a coordination layer for AI agents, what each subfolder is for, and where to look if a human wants to read the current state (answer: `index.md`). It does not need to be edited often.

## File: `_multi-agent/index.md`

The canonical "where things stand" page. Read by every agent at session start.

```yaml
---
last_updated: 2026-05-13 14:25:00
index_keeper: claude-research-pesaj
project_name: Pesaj 2027 materials
project_started: 2026-05-01
---
```

Body sections (in order):

1. **Project summary.** Two or three sentences. What is this project, who is it for, what is the deliverable.
2. **Current focus.** The two or three things actively being worked on right now, with wikilinks to task files.
3. **Agents and roles.** A list of registered agents and what each is doing. One line each.
4. **Open decisions.** Wikilinks to recent decision notes that affect ongoing work.
5. **Stale or blocked.** Anything that has not moved in more than a few days, with the reason.

Keep it under 100 lines. If it grows beyond that, content belongs in task files or decision notes, not in the index.

## File: `_multi-agent/events.md`

The shared append-only log. The single most important file in the protocol.

```yaml
---
file_type: events
rollover_policy: quarterly
current_period: 2026-Q2
---
```

Body: one event line per line, oldest at top, newest at bottom. No headers, no blank lines between entries, no edits to earlier lines under any circumstance.

### Event line grammar

```
- YYYY-MM-DD HH:MM | <agent-id> | <verb-phrase> | <wikilink-or-dash> | <note-or-dash>
```

Field-by-field:

- **Leading `- `**: standard markdown list marker. Lets Obsidian render the file cleanly.
- **Timestamp**: `YYYY-MM-DD HH:MM`. Local time in the vault's timezone. No seconds.
- **Agent ID**: the registered identifier of the writing agent. Verbatim, no quoting.
- **Verb phrase**: a short past-tense description of the action. Two to six words. Lowercase. Examples: `drafted outline`, `merged auth refactor`, `blocked on terminology`, `joined vault`, `took ownership`, `closed task`.
- **Wikilink**: a single wikilink to the most relevant note. If no specific note applies, use a single dash `-`.
- **Note**: an optional one-line elaboration. If absent, use a single dash `-`. No multi-line notes; longer context belongs in the agent's `log.md` or in a task file.

### Reserved verb phrases

These have special meaning and should be used consistently:

- `joined vault`: agent registration.
- `left vault`: agent retirement (rare; usually agents go silent rather than retire formally).
- `took ownership`: claimed a task that previously had no owner or a different owner (transfer should be paired with a handoff note).
- `closed task`: marked a task as done.
- `blocked on <something>`: cannot proceed without external input.
- `unblocked`: resumed after being blocked.
- `decided <slug>`: recorded a decision; the wikilink should point to the decision note.
- `assumed index keeper`: this agent is now the index keeper. Should follow an agreement or a release event from the previous keeper.
- `released index keeper`: this agent is no longer the index keeper. Usually paired with an `assumed index keeper` event from another agent in the same or nearby window.

Free-form verbs are encouraged for normal work; the reserved list is for state transitions that other agents may want to filter on.

## File: `_multi-agent/agents/<agent-id>/profile.md`

Static information about this agent. Updated only when role or capabilities change.

```yaml
---
agent_id: claude-research-pesaj
model: claude-opus-4-7
tool: claude.ai web
role: research and source-checking
languages: [es, en, he]
registered: 2026-05-01 09:12:00
---
```

Body: a few paragraphs of context. What does this agent do well? What is it bad at? What are its standing instructions? This is what other agents read to decide whether to hand work off to this one.

## File: `_multi-agent/agents/<agent-id>/status.md`

The agent's current state. Updated whenever the agent picks up or drops a task, or hits a blocker. Only the agent itself writes here.

```yaml
---
agent_id: claude-research-pesaj
last_active: 2026-05-13 14:22:00
state: working
current_task: tasks/literature-review
blockers: []
last_seen_event: 2026-05-13 14:20
---
```

`state` is one of: `working`, `idle`, `blocked`, `away`. When the agent has no active task, `current_task` is `null` (not the dash character or an empty string). Body is optional; if present, it is a one or two sentence note about what is in progress.

`last_seen_event` is the timestamp of the most recent event line this agent has processed. Used during the entry routine to scan only new events.

## File: `_multi-agent/agents/<agent-id>/inbox.md`

Append-only file for messages from other agents to this one.

```yaml
---
file_type: inbox
owner: claude-research-pesaj
---
```

Body: one entry per message, separated by `---` horizontal rules. Each entry has a header line with timestamp and sender, then a short body, then optionally a wikilink to context.

```markdown
### 2026-05-13 14:25 from claude-code

Frontend is ready for the Spanish translations; please prioritize `tasks/spanish-rollout`. Decision on "madrij" vs "leader" pending; see [[decisions/2026-05-13-vocabulary-madrij]].

---

### 2026-05-13 15:01 from gpt-translator-es

Wrapped first pass on the welcome packet. Two ambiguous phrases flagged in [[tasks/spanish-rollout]] under "Open questions".
```

The recipient is responsible for reading and acting on messages, and archiving them when done (see `references/troubleshooting.md`, section "Inbox bloat").

## File: `_multi-agent/agents/<agent-id>/log.md`

This agent's private journal. Audience: future instances of this same agent.

```yaml
---
file_type: log
owner: claude-research-pesaj
---
```

Body: free-form, append-only by convention but not by rule. Use it for reasoning traces, false starts, half-formed ideas, links to external resources, and anything that would clutter `events.md` if posted there. The format inside is up to the agent; suggested pattern is timestamped sections.

## File: `_multi-agent/tasks/<task-slug>.md`

One task per file. The slug is a short hyphenated identifier (e.g., `literature-review`, `spanish-rollout`, `jwt-rotation`).

```yaml
---
task_id: literature-review
title: Literature review for Pesaj sources
owner: claude-research-pesaj
status: in_progress
created: 2026-05-01 10:00:00
due: 2026-05-20
links: [tasks/source-database, decisions/2026-05-03-citation-format]
---
```

`status` is one of: `proposed`, `in_progress`, `blocked`, `done`, `abandoned`.

Body sections (in order):

1. **Goal.** One paragraph: what is "done" for this task?
2. **Current state.** Updated by the owner as work progresses. The single source of truth on this task's status.
3. **Open questions.** Bullet list. Where the task is uncertain. Other agents reading the task should look here first to see if they can help.
4. **Work log.** A narrative of what's been done, with wikilinks to specific outputs. Less granular than the agent's own `log.md`.
5. **Handoff history.** If ownership has changed, list each transfer with date and reason.

## File: `_multi-agent/decisions/YYYY-MM-DD-<slug>.md`

A lightweight ADR (architecture decision record), adapted for any decision worth referencing later.

```yaml
---
decision_id: 2026-05-13-vocabulary-madrij
title: Use "madrij" instead of "leader" in Spanish-language materials
decided_by: claude-research-pesaj
decided_on: 2026-05-13
affects_tasks: [tasks/spanish-rollout, tasks/welcome-packet]
status: accepted
---
```

`status` is one of: `proposed`, `accepted`, `superseded`. If superseded, add a `superseded_by:` field pointing to the newer decision.

Body sections:

1. **Context.** What was the situation that required a decision.
2. **Options considered.** A short list of alternatives.
3. **Choice.** Which option was picked.
4. **Consequences.** What changes as a result; what is now committed; what is now harder.

Decision notes are immutable in spirit; they should not be rewritten after being marked accepted. If the decision turns out to be wrong, write a new decision that supersedes the old one.

## File: `_multi-agent/handoffs/YYYY-MM-DD-HHMM-<from>-to-<to>.md`

A handoff note from one agent to another. Filename uses the timestamp of the handoff start.

```yaml
---
handoff_id: 2026-05-13-1430-claude-research-to-gpt-translator
from: claude-research-pesaj
to: gpt-translator-es
task: tasks/spanish-rollout
date: 2026-05-13 14:30:00
---
```

Body sections (in order):

1. **What's done.** Bullet list of completed work, with wikilinks.
2. **What's next.** Bullet list of remaining work.
3. **Where the state lives.** Wikilinks to every file the recipient should read.
4. **Known issues and gotchas.** Anything subtle that bit the previous owner.
5. **Recommended first step.** One concrete sentence: "Start by reading X and then doing Y."

The recipient does not edit the handoff note; instead they post an event line acknowledging it, optionally followed by a note in the previous owner's inbox.

## Naming and ID conventions, in one place

- **Agent IDs**: lowercase, hyphens, no underscores. Format: `<tool>-<role>-<project>`. The `<tool>` prefix (`claude`, `codex`, `gemini`, etc.) is conventionally required when the vault is shared across tools, because it lets any agent reading the event log see at a glance who produced which line. Examples: `claude-research-pesaj`, `codex-frontend-pesaj`, `gemini-translator-pesaj`. For single-tool vaults the prefix is optional but still recommended for future-proofing. Length: 8 to 40 characters.
- **Task slugs**: lowercase, hyphens. Length: 3 to 40 characters. Descriptive of the task, not the owner.
- **Decision slugs**: prefixed by date `YYYY-MM-DD-`, then a hyphenated description. Length of description: 3 to 40 characters.
- **Handoff filenames**: `YYYY-MM-DD-HHMM-<from-agent-id>-to-<to-agent-id>.md`.
- **Timezone**: by default UTC. Override in `_multi-agent/README.md` if the project has a strong reason to use local time (and document which local time).
